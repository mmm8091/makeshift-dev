# 草台编子识字班产品技术方案

## 目标

草台编子识字班第一版不是单纯文章站，而是一个带用户系统、卡密解锁、课程内容、作业展示和极简论坛的课程社区。

核心气质：

- 平民编程、AI 编程、草根互助。
- 从第一课开始让学员接触 GitHub PR 流程。
- 内容可持续写，系统可持续开发。
- 部署链路尽量 serverless，优先使用 Cloudflare 生态。

## 第一版范围

第一版必须包含：

- 公开首页。
- 公开课程前言。
- 登录/注册。
- 注册时填写昵称、GitHub 用户名、QQ 号。
- 卡密兑换。
- 除前言外的课程内容需要卡密解锁。
- 极简论坛，未解锁用户不可访问。
- 首页学员表。
- 学员通过 GitHub PR 把自己加入学员表。
- QQ 头像代理，头像可短缓存并自动更新。
- 代码提交后自动构建和部署。

第一版暂不做：

- 完整 LMS。
- 复杂积分/等级。
- 在线支付闭环。
- 富文本重论坛。
- 作业自动批改。

## 推荐技术栈

- 框架：Next.js
- 公开内容：MDX + Fumadocs
- 付费课程内容：Cloudflare D1 存 Markdown/元数据，服务端鉴权后读取和渲染
- 部署：Cloudflare Workers
- 数据库：Cloudflare D1
- ORM：Drizzle
- 用户系统：Better Auth
- UI：shadcn/ui + Tailwind CSS
- 论坛：自研极简论坛
- 评论：可选 giscus，仅用于公开文章或课程页评论
- 头像：Cloudflare Worker 代理 QQ 公开头像

选择理由：

- Next.js 适合同时做内容站和会员系统。
- Fumadocs 适合公开前言、说明文档、课程目录、代码块、搜索。
- 付费课程正文不能进入开源仓库、前端 bundle 或公开静态产物，必须由服务端鉴权后从 D1 读取。
- D1 足够支撑用户、卡密、论坛、学员表这些关系数据。
- Cloudflare Workers + D1 + Pages/Workers Git 集成能保持 serverless 部署体验。
- 自研极简论坛可以复用用户系统和卡密权限，不需要额外维护 Discourse/Flarum。

## 权限模型

访问层级：

- 游客：可看首页、前言、登录注册页、公开说明。
- 已注册未解锁用户：可登录、填写资料、兑换卡密，但不能看正式课程和论坛。
- 已解锁用户：可看课程、进论坛、发帖、回复、展示作业。
- 管理员：可生成卡密、禁用卡密、管理用户、删帖、置顶帖子。

课程访问规则：

- `/courses/preface` 公开。
- `/courses/**` 默认需要登录且拥有有效 entitlement。
- `/forum/**` 需要登录且拥有有效 entitlement。
- `/redeem` 需要登录。
- `/admin/**` 需要管理员权限。

## 内容存储与防泄漏策略

因为项目代码会开源，所以内容存储必须区分“公开内容”和“受保护内容”。

核心原则：

**需要卡密保护的正文，不能出现在公开 Git 仓库、前端 bundle、构建产物或可直接访问的静态文件里。**

如果把付费课程 Markdown/MDX 放在公开仓库里，再在页面上做登录或卡密拦截，本质上只能防普通用户，不能防会看源码的人。别人可以直接从 GitHub 或构建产物里拿到原文。

最终决策：

- 公开内容放仓库，用 MDX + Fumadocs 管理。
- 付费课程正文放 Cloudflare D1，用 Markdown 字段存储。
- 页面服务端读取课程前，先检查登录态和 entitlement。
- 未解锁用户只能看到公开目录、前言、试看摘要，不能拿到正文 API 响应。
- 论坛内容同样存 D1，并复用课程 entitlement 做访问控制。

内容分层：

```txt
公开仓库
- 首页文案
- 课程前言
- 招生说明
- 课程目录
- 试看文章
- 学员表 data/students.json
- 插画和公开设计资产

D1 数据库
- 付费课程正文 Markdown
- 课程章节元数据
- 卡密与权益
- 用户资料
- 论坛帖子与回复
```

第一版不使用私有 Git 仓库或 R2 存付费正文。它们可以作为未来选项，但 D1 的权限模型最直接，后台编辑、排序、发布状态也更好做。

推荐课程表：

```sql
course_sections
- id
- slug
- title
- summary
- body_md
- status
- visibility
- required_entitlement
- order_index
- published_at
- created_at
- updated_at

course_assets
- id
- section_id
- kind
- url
- alt
- order_index
- created_at
```

字段约定：

- `visibility = 'public'`：可公开访问，例如前言、试看。
- `visibility = 'locked'`：需要登录并拥有对应 entitlement。
- `status = 'draft'`：只有管理员可见。
- `status = 'published'`：按 visibility 和权限规则展示。
- `body_md` 只存 Markdown，不在公开 repo 里保留副本。

课程读取流程：

1. 用户访问课程页。
2. 服务端根据 slug 查询 `course_sections` 的元数据。
3. 如果 `visibility = 'public'`，返回正文。
4. 如果 `visibility = 'locked'`，检查用户 session。
5. 检查用户是否拥有 `required_entitlement`。
6. 权限通过后，服务端读取 `body_md` 并渲染。
7. 权限不通过时，只返回标题、摘要和解锁引导，不返回正文。

需要避免的做法：

- 不把付费正文写进 `content/courses/*.mdx`。
- 不在构建时把付费正文生成静态 HTML。
- 不把正文塞进前端 JSON 后再用 JS 隐藏。
- 不用纯前端路由守卫保护课程内容。
- 不把 D1 里的付费正文导出到公开仓库做备份。

## 卡密系统

卡密用于批量生成、对外销售、学员兑换。

设计原则：

- 数据库只存卡密 hash，不存明文。
- 卡密兑换用原子条件更新防并发重复使用（D1 无交互式事务，详见兑换流程）。
- 支持批量生成。
- 支持过期时间。
- 支持禁用。
- 支持单次使用或多次使用。
- 支持不同权益，例如 `course_full`、`cohort_1`、`lifetime`。

推荐表：

```sql
redeem_codes
- id
- code_hash
- batch_id
- entitlement_scope
- max_uses
- used_count
- expires_at
- disabled_at
- created_by
- created_at

redeem_code_uses
- id
- redeem_code_id
- user_id
- used_at
- ip_hash
- user_agent_hash

entitlements
- id
- user_id
- scope
- source
- source_id
- starts_at
- expires_at
- created_at
```

兑换流程（D1 无交互式事务，必须用原子条件更新防并发超发）：

1. 用户登录。
2. 输入卡密，服务端标准化格式并计算 hash。
3. 执行一条原子条件更新，只有同时满足「未禁用、未过期、未达使用上限」时才真正占用一次名额：

```sql
UPDATE redeem_codes
SET used_count = used_count + 1
WHERE code_hash = ?
  AND disabled_at IS NULL
  AND used_count < max_uses
  AND (expires_at IS NULL OR expires_at > ?);
```

4. 检查这条 UPDATE 实际影响的行数：
   - 影响 0 行：卡密不存在、已禁用、已过期或已用完，返回失败，不发放任何权益。
   - 影响 1 行：抢占成功，进入下一步。
5. 用 `db.batch()` 在同一隐式事务里写入 `redeem_code_uses` 记录，并写入或延长 `entitlements`。
6. 返回解锁成功。

说明：

- D1 不支持 `BEGIN/COMMIT` 交互式事务，「先查询、再判断、再写入」的顺序流程在并发下会超发，必须依赖单条 `UPDATE ... WHERE` 的原子性来抢名额。
- `used_count` 的自增和上限判断必须放在同一条 SQL 里，这是防并发重复使用的关键。
- 单次卡密与多次卡密共用同一套逻辑，单次卡密即 `max_uses = 1`。

卡密格式建议：

```txt
CTBZ-XXXX-XXXX-XXXX
```

其中 `CTBZ` 是“草台编子”的拼音缩写，方便识别和客服处理。

## QQ 头像代理

目标：

- 用户只填 QQ 号。
- 首页和论坛自动显示 QQ 头像。
- QQ 头像变更后，站点短缓存过期即可自动更新。
- 前端不直接暴露腾讯头像 URL 逻辑。

可用头像源：

```txt
https://q1.qlogo.cn/g?b=qq&nk={qq}&s=100
https://q.qlogo.cn/headimg_dl?dst_uin={qq}&spec=640&img_type=jpg
```

建议服务端接口：

```txt
GET /api/avatar/qq/:qq
```

规则：

- 只允许纯数字 QQ 号。
- QQ 号长度做基本限制，例如 5 到 12 位。
- 使用 `fetch` 请求腾讯头像。
- 设置 `Cache-Control: public, max-age=86400, stale-while-revalidate=604800`。
- 失败时返回默认头像。
- 可在 Cloudflare Cache API 或普通 HTTP 缓存里缓存。

示意逻辑：

```ts
const QQ_AVATAR_CACHE_SECONDS = 60 * 60 * 24;

export async function getQQAvatar(qq: string) {
  if (!/^\d{5,12}$/.test(qq)) {
    return defaultAvatar();
  }

  const url = `https://q1.qlogo.cn/g?b=qq&nk=${qq}&s=100`;
  const upstream = await fetch(url, {
    headers: {
      "User-Agent": "Mozilla/5.0",
    },
  });

  if (!upstream.ok) {
    return defaultAvatar();
  }

  return new Response(upstream.body, {
    headers: {
      "Content-Type": upstream.headers.get("Content-Type") ?? "image/jpeg",
      "Cache-Control": `public, max-age=${QQ_AVATAR_CACHE_SECONDS}, stale-while-revalidate=604800`,
    },
  });
}
```

## 首页学员表

首页用 QQ 头像墙展示学员：默认只显示头像，鼠标悬停时显示昵称。

每个学员在 `data/students.json` 里只需要两个核心字段：

- `qq`：用于显示 QQ 头像。
- `displayName`：昵称，鼠标悬停时显示。

字段刻意精简。第一版首页不展示 GitHub 用户名、学习宣言、PR 链接、作业链接，避免学员表过重；这些信息后续可以放到个人页或论坛，不挤占首页。GitHub 用户名仍在注册时收集并存入 D1 的 `profiles`，与首页学员表解耦。

github第一课作业：

1. 注册账号，填写 GitHub 用户名和 QQ 号。
2. Fork 课程仓库。
3. 修改 `data/students.json`，添加自己的 `qq` 和 `displayName`。
4. 提交 PR。
5. 合并后自动部署，首页头像墙出现自己的头像，悬停显示昵称。

学员表条目示例：

```json
{
  "qq": "12345678",
  "displayName": "某某"
}
```

注意：

- PR 模板里需要提示：提交 QQ 号意味着同意在首页公开展示，并用于显示 QQ 头像；该信息会进入公开仓库的 Git 历史，合并后无法完全撤回。
- 管理员审核 PR 时检查格式、内容、QQ 是否为纯数字。
- 首页渲染头像时不要直接拼腾讯 URL，统一走 `/api/avatar/qq/:qq`。

## 极简论坛

论坛第一版只做课程社区需要的功能。

功能：

- 帖子列表。
- 发帖。
- 回帖。
- 标签。
- 作业分享标签。
- 管理员置顶。
- 管理员删除/隐藏。
- 作者可编辑自己的帖子。

不做：

- 私信。
- 复杂通知。

- 大型附件系统。

权限：

## 内容结构

公开仓库只放公开内容和内容壳，不放付费正文。

建议仓库目录：

```txt
content/
  public/
    preface.mdx
    enrollment.mdx
    trial-lesson.mdx
  course-catalog/
    index.mdx
data/
  students.json
设计资产/
  前言/
```

付费课程正文存 D1，不进入上述目录。

课程目录可以公开，但目录只包含标题、摘要、顺序和解锁状态，不包含正文。

公开课程元数据示例：

```ts
{
  title: "第一次 PR：把自己加到学员表",
  slug: "01-github-pr",
  public: false,
  order: 1,
  summary: "通过一次真实 PR 完成课程社区的第一次参与",
  requiredEntitlement: "course_full"
}
```

前言设置：

```ts
public: true
```

其他课程默认：

```ts
public: false
requiredEntitlement: "course_full"
```

页面实现要点：

- 公开 MDX 可走 Fumadocs 静态渲染。
- 付费课程页走服务端路由或 server component。
- 课程正文 API 必须做 session + entitlement 检查。
- 未解锁时只返回课程摘要，不返回 `body_md`。
- 搜索功能第一版只索引公开内容；付费内容搜索后续单独做带权限的服务端搜索。

## 部署流水线

目标：

- push 到主分支自动部署生产。
- PR 自动生成预览地址。
- 构建状态回写 GitHub。
- 学员 PR 合并后首页自动更新。

推荐方式：

- Cloudflare Workers/Pages 连接 GitHub 仓库。
- 构建命令：`pnpm build`
- 部署命令：`pnpm deploy` 或 Cloudflare 自动部署配置。
- 数据库迁移由管理员在发布前执行，或通过受控 CI job 执行。

环境：

```txt
BETTER_AUTH_SECRET
BETTER_AUTH_URL
GITHUB_CLIENT_ID
GITHUB_CLIENT_SECRET
CLOUDFLARE_ACCOUNT_ID
CLOUDFLARE_API_TOKEN
D1_DATABASE_ID
```

建议 scripts：

```json
{
  "dev": "next dev",
  "build": "next build",
  "preview": "opennextjs-cloudflare build && opennextjs-cloudflare preview",
```

## 安全和风控

必须做：

- 卡密 hash 存储。
- 兑换接口限流。
- 发帖接口限流。
- 付费课程正文不得进入公开仓库、前端 bundle 或静态构建产物。
- 付费课程正文 API 必须服务端鉴权，不允许纯前端隐藏。
- QQ 号格式校验。
- Markdown 渲染 XSS 防护。
- 管理员后台加权限检查。
- PR 模板说明公开 QQ 的含义。

建议做：

- Turnstile 防机器人注册。
- 用户可修改或删除 QQ 号。
- 管理员可禁言用户。
- 论坛帖子软删除，不直接硬删。
- 卡密批次管理，方便追踪销售渠道。

## 第一版判断

第一版最重要的不是功能多，而是把三件事跑通：

1. 学员能买卡密并解锁课程。
2. 学员能通过 PR 把自己放到首页。
3. 学员能在论坛发作业、互相看见、互相加 QQ。

这三个跑通以后，草台就真的搭起来了。
